Architecture & Log Pipelines

Two independent pipelines ship logs from each node to Loki. They capture different event sets and have different failure modes.

Pipeline 1 — OS Syslog
systemd journal → rsyslog → /var/log/syslog → Promtail → Loki

Covers: syslog, kern.log, auth.log.
Job label: job="syslog"

Pipeline 2 — systemd Journal
systemd journal → Promtail journal input → Loki

Filtered: warning+ severity and all logind events.
Job label: job="journal"

Why two pipelines? rsyslog rate-limits high-frequency events before they reach syslog. During the puffer ACPI incident (#59), 312 Power key pressed events were silently dropped by rsyslog and never appeared in Loki. The journal pipeline reads the binary journal directly — no rate limiting, and cursor-based recovery ships pre-crash entries after a reboot.

Query Tools

just loki-syslog-node syslog pipeline

Queries {job="syslog", node="<node>"} — covers syslog, auth.log, and kern.log together.

just loki-syslog-node <node> [since] [filter]

just loki-syslog-node whale          # last 1 hour
just loki-syslog-node mullet 6h      # last 6 hours
just loki-syslog-node tuna 24h error # last 24h, grep "error"

Includes a 5-minute stream health check at the end.

just loki-journal-check journal pipeline

Queries {job="journal", node="<node>"} — warning+severity events and all logind events. Use for crashes, OOM, power events, unit failures.

just loki-journal-check <node> [filter] [since]

just loki-journal-check puffer "Power key" 24h
just loki-journal-check gold "OOM" 48h
just loki-journal-check whale "entered failed state" 7d

Includes a 5-minute stream health check at the end.

just loki-node-events content search

Searches syslog content for the node name. Reaches pre-2026-05-12 data that lacks the node label. Also available: loki-shutdown-events, loki-idrac, loki-reboot-history, loki-tail.

just loki-node-events puffer 24h
just loki-shutdown-events puffer 7d
just loki-idrac puffer 30d
just loki-reboot-history 7d

Common Scenarios

Incident Node rebooted unexpectedly

Check boot history, then look at journal for what happened right before shutdown.

just loki-reboot-history 7d                       # all nodes: boot times + shutdown count
just loki-shutdown-events puffer 7d               # power/reboot events on specific node
just loki-journal-check puffer "shutdown\|reboot\|panic" 24h  # pre-crash journal entries

The journal pipeline uses cursor-based recovery — if Promtail was running before the crash, pre-reboot entries will appear in Loki after the node comes back up.

Incident Pod evicted due to disk pressure (mullet)

Mullet's root disk (/dev/sda5, 117G) hit DiskPressure in 2026-05 (issue #77). Check disk-related syslog and kubelet events.

just loki-syslog-node mullet 6h "disk\|evict\|pressure"
just loki-journal-check mullet "kubelet\|DiskPressure" 6h

Monitor root disk usage: Prometheus alert fires at 80% (MulletRootDiskHigh).

Investigate Syslog missing from a node in Loki

Run the chainsaw test first, then check Promtail drop metrics.

just test-loki                                    # loki-healthy + promtail-drop-rate-low
kubectl get pods -n loki -l app.kubernetes.io/name=promtail -o wide  # pod per node?
# Check Prometheus for permanent drops:
# rate(promtail_dropped_entries_total{reason="ingester_error"}[1h]) * 3600

If a pod is dropping >500 entries/hour, the chainsaw test will fail. Restart the Promtail pod to clear state; investigate Loki ingester health if drops recur.

Tip: promtail-drop-rate-low is an early warning — it fires before syslog-from-all-nodes notices a gap. If drops are >500/hr but syslog is still flowing, Promtail is under stress but hasn't lost data yet.

Known Issues

Open Periodic Loki ingester 500s — every few hours, all nodes simultaneously

Loki returns rpc error: code = DeadlineExceeded desc = context deadline exceeded on Promtail push at irregular intervals (roughly every 4–6 hours). All nodes are affected at the same timestamp, suggesting an internal Loki event (WAL flush, compaction, or ingester backpressure from Ceph S3 write latency). Promtail retries and usually recovers; the promtail-drop-rate-low chainsaw test will catch it if drops become acute (>500/hr).

Tracked in issue #89. Root cause investigation pending.

Resolved 2026-05-12 Whale syslog absent from Loki for 7 days — issue #89

Root cause: replacement: ${NODE_NAME} in Promtail relabel_configs was treated as a regex capture group reference (not an env var), silently producing an empty label. All syslog landed in one unlabeled shared stream. Fixed with a pipeline_stages regex that extracts the hostname from syslog content. Two timestamp formats handled:

# Traditional (Ubuntu 22.04 — whale):
May 12 12:34:53 whale systemd[1]: ...

# RFC 3339 (Ubuntu 24.04 — all other nodes):
2026-05-12T12:34:53.47-04:00 mullet caddy[2443]: ...
