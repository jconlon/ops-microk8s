Object Storage Design

Architecture design for Ceph RGW S3 object storage — dated 2026-01-05

Implemented: Option 2 (Erasure Coding ec:2+2) was chosen and is live at s3.verticon.com. This document records the design rationale and trade-offs evaluated at the time.

Design Options

Option 1 — Replicated (3×)
Usable capacity: ~5.3 TB
Failure tolerance: 1 OSD
Min OSDs: 3
  • Simpler config, lower CPU
  • Consistent with block storage
  • Better for small objects
  • 33% storage efficiency
Option 2 — Erasure Coding ec:2+2 Chosen
Usable capacity: ~8 TB
Failure tolerance: 2 OSDs
Min OSDs: 4
  • 50% storage efficiency (vs 33%)
  • Higher durability
  • Suited for large objects (backups, media)
  • Higher CPU overhead for encode/decode

Scalability

Adding drives to an existing Ceph cluster is non-disruptive: update the CephCluster manifest to include new devices, Rook creates new OSDs, and Ceph rebalances automatically with no downtime.

EC scheme is immutable after pool creation. Changing from ec:2+2 to ec:4+2 requires creating a new pool, migrating all data, and updating RGW — plan the scheme before writing significant data.
