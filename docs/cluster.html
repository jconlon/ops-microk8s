Networking

LoadBalancer — MetalLB

IP range: 192.168.0.200–192.168.0.230. Each service gets a fixed IP via metallb.universe.tf/loadBalancerIPs annotations.

Avoid conflicts: The PiHole DNS server is on the same subnet. New MetalLB IPs must not overlap with DHCP reservations or PiHole's address.

External DNS + TLS — kgateway + cert-manager (issue #108)

All in-scope HTTP(S) cluster hostnames follow the same pattern — see networking.html for the full hostname table:

  • Cloudflare A record: <service>.verticon.com → shared kgateway MetalLB IP 192.168.0.224 (DNS only, not proxied)
  • kgateway (Envoy-based Gateway API): one shared Gateway with a per-hostname HTTPS listener, SNI-routed
  • cert-manager issues real Let's Encrypt certs per hostname via DNS-01 through Cloudflare — HTTP-01 can't work since *.verticon.com resolves to private LAN IPs
  • Caddy on mullet no longer fronts any cluster service — it only serves mullet's own static sites (mullet.verticon.com, mullet.verticon.lab)
# Adding a new hostname requires 3 resources (see README "Adding a New DNS Name for a Service"):
# 1. Certificate           — cert-manager-gitops/resources/certificates/<service>-certificate.yaml
# 2. HTTPS listener        — kgateway-gitops/resources/gateway.yaml
# 3. HTTPRoute             — kgateway-gitops/resources/httproutes/<service>-httproute.yaml

CI/CD Pipeline

CI is event-driven: a git push on any application repo triggers an image build via Argo Events → Argo Workflows → Kaniko → Harbor.

git push origin main
  → post-push git hook (developer machine, LAN)
    → HTTP POST → Argo Events WebhookEventSource (events.verticon.com)
      → Sensor → Argo Workflows
        → image-build-push ClusterWorkflowTemplate
          → Kaniko build → Harbor registry
            → create-pr step updates image tag in spec repo

Argo Workflowsargo-workflowsworkflows.verticon.com · 192.168.0.209:2746
Argo Events webhookargo-eventsevents.verticon.com · 192.168.0.221:12000

Reusable templates

# Build any image:
argo submit -n argo-workflows \
  --from clusterworkflowtemplate/image-build-push \
  -p repo-url=https://github.com/jconlon/<repo> \
  -p image=registry.verticon.com/<project>/<name>:<tag> \
  -p context=<subdir>

Secrets management — Teller

Cluster K8s secrets are managed out-of-band via Teller configs in teller/. Secrets are sourced from Google Secret Manager.

teller run --config teller/.teller-freshrss.yml -- bash -c 'kubectl apply ...'
teller run --config teller/.teller-postgresql.yml -- bash -c 'kubectl apply ...'

Node drain / maintenance

kubectl drain <node> --ignore-daemonsets --delete-emptydir-data
# ... perform maintenance ...
kubectl uncordon <node>
