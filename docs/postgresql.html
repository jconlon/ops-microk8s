v_freshrss_entries view

A denormalized read view joining entry → feed → category → tag for the admin user. Consumers query this view instead of joining the normalized tables directly.

Fan-out note: The view uses a LEFT JOIN on tags. If an entry has multiple tags it will appear more than once — one row per tag. Currently all entries have at most one tag (29 tag assignments across 11 k+ entries), so the view is effectively 1-to-1 with the entry table. Run just test-freshrss to assert this invariant holds. 
SELECT e.*, f.name AS feed_name, f.url AS feed_url, ...,
       c.id AS category_id, c.name AS category_name, ...,
       t.id AS tag_id, t.name AS tag_name, ...
FROM freshrss_admin_entry e
  JOIN  freshrss_admin_feed     f  ON f.id = e.id_feed
  LEFT JOIN freshrss_admin_category c  ON c.id = f.category
  LEFT JOIN freshrss_admin_entrytag et ON et.id_entry = e.id
  LEFT JOIN freshrss_admin_tag  t  ON t.id = et.id_tag;

Chainsaw coverage: tests/freshrss/chainsaw-test.yaml — run with just test-freshrss. Checks: view has rows, no orphaned feed/category joins, row count ≥ raw entry count, no unexpected fan-out, required columns non-null.
