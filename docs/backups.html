Overview

Three independent strategies cover different data categories. All targets land in Ceph RGW (s3.verticon.com / 192.168.0.204) — the cluster's internal S3-compatible object store.

StrategyWhatBucketRetentionMechanism
Restic /home/jconlon (excluding Music, Downloads, caches) restic-backups 24h · 7d · 4w · 6m · 2y Systemd timer on mullet, daily 3 AM
CloudNativePG All PostgreSQL databases (WAL + base backups) postgresql-backups 30 days Barman continuous streaming from primary
mc mirror ~/Music and ~/Pictures/pictures music-library · pictures Permanent (mirror) Systemd timer on mullet, nightly
All backups use Ceph RGW as the backend. If the Ceph cluster is degraded, new backups will fail but existing snapshots remain intact — Ceph stores 3× replicated copies across the four Dell R320 storage nodes (gold, squid, puffer, carp).

Restic — Home Directory

Restic backs up /home/jconlon to s3:http://192.168.0.204/restic-backups. Credentials are injected at runtime via Teller from Google Secret Manager — no secrets in the repo or systemd units.

What is backed up

  • All of /home/jconlon — dotfiles, git repos, documents, config
  • Excluded: ~/Music (covered by mc mirror), ~/Downloads, caches (.cache, node_modules, .venv, build artifacts), and ~/Pictures/pictures (covered by mc mirror)

Systemd services

All three services live in scripts/restic/systemd/. Install once with sudo ./install-restic.sh.

ServiceScheduleWhat it doesLog
restic-backup.timer Daily · 3:00 AM Incremental snapshot of /home/jconlon to Ceph RGW /var/log/restic-backup.log
restic-prune.timer Weekly · Sun 4:00 AM Enforce retention policy, remove old snapshots, reclaim space /var/log/restic-prune.log
restic-verify.timer Monthly · 1st 5:00 AM Integrity check (reads 5% of data), lists 5 most recent snapshots /var/log/restic-verify.log
Persistent timers: All timers use Persistent=true. If mullet was off at the scheduled time, the job runs automatically on next boot (after the OnBootSec delay). No missed backups due to downtime.

Ad-hoc commands

# Check timer status and next run
systemctl list-timers restic-*
just backup-status

# Run a backup immediately
sudo systemctl start restic-backup.service

# Follow backup logs
sudo journalctl -u restic-backup.service -f
tail -f /var/log/restic-backup.log

PostgreSQL — Continuous WAL Archiving

CloudNativePG streams WAL (Write-Ahead Log) continuously to Ceph RGW and takes periodic base backups. This enables point-in-time recovery (PITR) to any second within the retention window.

Configuration

SettingValue
Destinations3://postgresql-backups/production/
S3 endpointhttp://rook-ceph-rgw-rook-ceph-rgw.rook-ceph.svc:80 (internal)
Retention30 days
WAL compressiongzip · 2 parallel streams
Data compressiongzip · 2 parallel jobs
Backup targetPrimary
Credentials secretceph-s3-credentials in postgresql-system namespace
Barman Cloud: CNPG uses barman-cloud-wal-archive and barman-cloud-backup under the hood. WAL segments are archived continuously as transactions commit — recovery is possible to any point in the last 30 days, not just the last base backup.

Status and management

# Cluster health and backup status
just pg-status
kubectl cnpg status production-postgresql -n postgresql-system

# List available backups
kubectl cnpg backup list production-postgresql -n postgresql-system

# Trigger a base backup immediately
kubectl cnpg backup production-postgresql -n postgresql-system

# Check WAL archiving (should show 0 failed)
kubectl cnpg status production-postgresql -n postgresql-system | grep -A5 "WAL"

Recreating the credentials secret

If the ceph-s3-credentials secret is lost or rotated:

teller run --config teller/.teller-postgresql.yml -- bash -c '
  kubectl create secret generic ceph-s3-credentials \
    --namespace postgresql-system \
    --from-literal=ACCESS_KEY_ID="$ACCESS_KEY_ID" \
    --from-literal=ACCESS_SECRET_KEY="$ACCESS_SECRET_KEY" \
    --dry-run=client -o yaml | kubectl apply -f -
'

Media Sync — Music & Pictures

Two nightly mc mirror jobs keep the media libraries in sync with Ceph RGW. These are mirrors, not versioned backups — deleted local files will eventually be deleted from the bucket too.

Music Library

~72 GB · ~3,200 objects

Source~/Music
Bucketceph-rgw/music-library
ScheduleDaily · 2:00 AM
Log/var/log/music-sync.log
Pictures

Growing

Source~/Pictures/pictures
Bucketceph-rgw/pictures
ScheduleDaily · 3:00 AM
Log/var/log/pictures-sync.log
Mirror semantics: mc mirror is one-way and non-versioned. If a file is deleted locally it will be deleted from the bucket on the next sync. For versioned protection of the music library, use Restic (which backs up /home/jconlon but excludes ~/Music). Consider enabling Ceph RGW object versioning on these buckets if accidental deletion is a concern.

Restore Procedures

Restore a file from Restic

# In the dotfiles devbox context, run all restic commands via teller:
cd ~/dotfiles

# Find the snapshot containing the file
devbox run -- teller run \
  --config /home/jconlon/dotfiles/restic/restic/.teller-restic-ceph.yml \
  -- restic snapshots

# Search for a specific file across all snapshots
devbox run -- teller run \
  --config /home/jconlon/dotfiles/restic/restic/.teller-restic-ceph.yml \
  -- restic find --long /home/jconlon/path/to/file

# Restore a single file from a snapshot (use snapshot ID from above)
devbox run -- teller run \
  --config /home/jconlon/dotfiles/restic/restic/.teller-restic-ceph.yml \
  -- restic restore <snapshot-id> \
     --include /home/jconlon/path/to/file \
     --target /tmp/restore

# Restore entire home directory from latest snapshot
devbox run -- teller run \
  --config /home/jconlon/dotfiles/restic/restic/.teller-restic-ceph.yml \
  -- restic restore latest --target /tmp/restore

PostgreSQL point-in-time recovery (PITR)

CNPG supports recovering to a specific timestamp within the 30-day retention window. Create a new cluster pointing at the backup and specify the recovery target:

# Create a recovery cluster from the backup (edit YAML with desired targetTime)
kubectl apply -f - <<EOF
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: production-postgresql-recovery
  namespace: postgresql-system
spec:
  instances: 1
  bootstrap:
    recovery:
      source: production-postgresql
      recoveryTarget:
        targetTime: "2026-05-10 12:00:00"  # adjust as needed
  externalClusters:
    - name: production-postgresql
      barmanObjectStore:
        destinationPath: "s3://postgresql-backups/production/"
        endpointURL: "http://rook-ceph-rgw-rook-ceph-rgw.rook-ceph.svc:80"
        s3Credentials:
          accessKeyId:
            name: ceph-s3-credentials
            key: ACCESS_KEY_ID
          secretAccessKey:
            name: ceph-s3-credentials
            key: ACCESS_SECRET_KEY
EOF
PITR is a new cluster, not an in-place restore. The recovered cluster comes up in read-only mode until you promote it. Validate data before switching application connections.
