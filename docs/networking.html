Networking

How Cloudflare DNS, kgateway (Gateway API), Caddy, and MetalLB route external traffic to cluster services

Update (issue #108): the migration from one-MetalLB-IP-per-service + Caddy to a single shared kgateway (Gateway API) IP is complete for all in-scope HTTP(S) services (15 total). Caddy on mullet no longer fronts any cluster service — it now only serves its own static-file sites (mullet.verticon.com/.lab), which have no Kubernetes backend and can't move to kgateway without a redesign. New HTTP(S) services should always use the kgateway path — see "Which path does a service use?" below.

Overview

All *.verticon.com hostnames are real, public DNS records — but they resolve to private LAN IPs (mullet's LAN IP, or the shared kgateway IP), not a public internet address. This is intentional: these services are only meant to be reachable from inside the LAN (or via VPN), while still using real hostnames and real Let's Encrypt certificates. Because the IPs are never reachable from the public internet, certificate issuance for both paths uses the ACME DNS-01 challenge (proving domain ownership via a Cloudflare DNS TXT record) — HTTP-01 could never succeed here, since Let's Encrypt's validators can't open a connection to a private IP no matter what the DNS record points to.

Path 1 — kgateway (all in-scope HTTP(S) services)

Internet client │ │ DNS query: grafana.verticon.com ▼ Cloudflare DNS ────────────────────────────── A record → 192.168.0.224 (shared kgateway IP) │ │ HTTPS :443, SNI = grafana.verticon.com ▼ kgateway Gateway "cluster-gateway" ── TLS termination (Let's Encrypt cert via cert-manager DNS-01/Cloudflare) │ │ HTTPRoute match on hostname → backend Service ▼ Grafana Service (monitoring/grafana) │ ▼ Grafana pod (inside cluster)

Path 2 — Caddy on mullet (static sites only — no cluster service uses this anymore)

Internet client │ │ DNS query: mullet.verticon.com ▼ Cloudflare DNS ────────────────────────────── A record → mullet's LAN IP │ │ HTTPS :443 ▼ mullet (Caddy) ── TLS termination (Let's Encrypt cert via Cloudflare DNS-01) │ │ file_server, root * /var/caddy/site ▼ Static files on mullet's local disk (no Kubernetes involved at all)

This is the only remaining use of Caddy — serving its own static sites directly from mullet's filesystem. There's no Kubernetes Service for these, so there's nothing for an HTTPRoute to route to; migrating them would mean first standing up an in-cluster static-file service.

Components

Cloudflare DNS

DNS records are managed at dash.cloudflare.com → verticon.com → DNS → Records. Every service has an A record set to DNS-only (grey cloud — proxying disabled), pointing at either 192.168.0.224 (kgateway — all 15 in-scope HTTP(S) services) or mullet's LAN IP (Caddy — only its own static-file sites now). Cloudflare proxying is intentionally off for both paths.

Cloudflare is also the ACME DNS provider for both kgateway/cert-manager and Caddy — each creates a temporary _acme-challenge TXT record via the Cloudflare API to prove domain ownership during certificate issuance, using separate API tokens (see issue #109) so they can be rotated/revoked independently.

kgateway (Gateway API)

kgateway (CNCF sandbox, Envoy-based, formerly Gloo Gateway) runs in the kgateway-system namespace. A single Gateway resource (cluster-gateway) holds one shared HTTP listener (port 80, redirect-only — not used for ACME) plus one HTTPS listener per migrated hostname (SNI-routed, each with its own tls.certificateRefs). One HTTPRoute per service attaches to the matching listener and routes to that service's backend Service.

cert-manager (cert-manager namespace) issues the certificates: a ClusterIssuer (letsencrypt-http01 — name predates the DNS-01 correction, the mechanism is DNS-01) with a dns01.cloudflare solver, referencing a Cloudflare API token Secret (cloudflare-api-token-secret, bootstrapped via teller/.teller-cert-manager.yml). One Certificate resource per hostname.

Caddy on mullet (remaining services)

Caddy is installed as a systemd service on mullet (Ubuntu 22.04). The configuration lives at /etc/caddy/Caddyfile — not managed by this repo's GitOps. Caddy uses the caddy-dns/cloudflare plugin (compiled into the binary) to handle its own DNS-01 challenges.

MetalLB

MetalLB assigns each Kubernetes LoadBalancer service a stable IP from the pool 192.168.0.200–192.168.0.230. These IPs are on the cluster LAN and are only reachable from mullet and other LAN hosts — they are never directly exposed to the internet. 192.168.0.224 is the single shared IP for the kgateway Gateway's Service; every other IP in the pool is still one-per-service (Caddy-fronted HTTP(S) services, or raw TCP services like PostgreSQL/S3/Kafka that don't go through either proxy).

Adding a new service (current procedure)

For a new HTTP(S) service, use kgateway:

  1. Confirm the backend Service. Don't assume name/namespace/port — check:
    kubectl get svc -A | grep <service>
  2. Add a Certificate in cert-manager-gitops/resources/certificates/<service>-certificate.yaml (namespace kgateway-system, issuerRef: letsencrypt-http01).
  3. Add an HTTPS listener to kgateway-gitops/resources/gateway.yaml's shared Gateway, named https-<service>, scoped to that hostname.
  4. Add an HTTPRoute in kgateway-gitops/resources/httproutes/<service>-httproute.yaml, in the same namespace as the backend Service (avoids needing a ReferenceGrant).
  5. Add a Cloudflare DNS A record pointing at 192.168.0.224 (DNS only, not proxied) — same dashboard steps as before, just a different target IP.
  6. Commit and push. ArgoCD auto-syncs (directory.recurse: true is required on both resources apps). Verify:
    curl --resolve <service>.verticon.com:443:192.168.0.224 https://<service>.verticon.com/
Two gotchas found the hard way: (1) kgateway routes HTTPS listeners by TLS SNI, not the HTTP Host header — curl -H "Host: ..." against the raw Gateway IP gets a TLS-level connection reset; use --resolve instead. (2) This machine's local DNS resolver caches the old A record for its TTL after a Cloudflare change — a check run here right after a cutover may show a stale answer for a minute or two. Public resolvers (dig @1.1.1.1) reflect the change immediately; real end users are unaffected since their own resolvers were never warmed with the stale answer.

For a new non-HTTP(S) / raw TCP service (PostgreSQL, S3, Kafka's external listener, etc.), or an HTTP(S) service that for some reason still needs the old path, use Caddy:

  1. Assign a MetalLB IP. Pick the next free IP from the pool (192.168.0.200–230). Check current assignments with:
    kubectl get svc -A --field-selector spec.type=LoadBalancer
  2. Annotate the Kubernetes service. Add the MetalLB annotation to the service manifest: 
    metadata:
  annotations:
    metallb.universe.tf/loadBalancerIPs: "192.168.0.NNN"
  3. Add a Cloudflare DNS A record pointing at mullet's LAN IP, DNS only.
  4. Add a Caddyfile block on mullet at /etc/caddy/Caddyfile: 
    <service>.verticon.com {
    reverse_proxy http://192.168.0.NNN:80
    tls {
        resolvers 1.1.1.1 1.0.0.1
    }
}
  5. Reload Caddy:
    sudo systemctl reload caddy
    Caddy will automatically request a TLS certificate for the new hostname via DNS-01.
  6. Update docs/index.html — add the service to the Web UIs or Service Endpoints section.

Troubleshooting

kgateway: HTTPRoute not reachable

  • Check Gateway, HTTPRoute, and Certificate status: kubectl get gateway,httproute -A, kubectl get certificate -n kgateway-system
  • Test with the correct SNI, not just a Host header: curl -v --resolve <host>:443:192.168.0.224 https://<host>/
  • Check the Certificate's ACME chain if stuck pending: kubectl get certificaterequest,order,challenge -n kgateway-system
  • Confirm the Cloudflare API token Secret exists and has both Zone:DNS:Edit and Zone:Zone:Read (the latter is needed to resolve the zone name to a zone ID — easy to miss)

kgateway: local check fails right after a DNS cutover

  • Confirm the record actually changed publicly: dig @1.1.1.1 +short <host> (bypasses any local cache)
  • If public DNS is correct but a local curl/dig still shows the old IP, it's local resolver cache lag (systemd-resolved) — verify against the real target with --resolve instead of waiting

Caddy: certificate not issued / renewal failed

  • Check Caddy logs: journalctl -u caddy -f
  • Confirm the CLOUDFLARE_API_TOKEN env var is set and the token has Zone DNS Edit permissions for verticon.com
  • Confirm the Cloudflare A record exists and is set to DNS-only (not proxied)
  • Caddy caches cert failures — after fixing, run sudo systemctl restart caddy to retry immediately

Caddy: service unreachable after adding a Caddyfile block

  • Verify the MetalLB service has an external IP: kubectl get svc -n <ns> <name>
  • Test the LAN target directly from mullet: curl -v http://192.168.0.NNN:80
  • Confirm Caddy reloaded cleanly: sudo systemctl reload caddy && journalctl -u caddy -n 30

Validate Caddyfile syntax before reload

caddy validate --config /etc/caddy/Caddyfile
