Networking

How Cloudflare DNS, Caddy, and MetalLB route external traffic to cluster services on mullet

Overview

All *.verticon.com hostnames resolve to mullet's WAN IP via Cloudflare DNS. Caddy runs on mullet as a TLS-terminating reverse proxy, forwarding each hostname to the appropriate MetalLB LoadBalancer IP inside the cluster LAN (192.168.0.x). Certificates are obtained automatically via the ACME DNS-01 challenge using the Cloudflare API — no public HTTP exposure is required.

Internet client │ │ DNS query: grafana.verticon.com ▼ Cloudflare DNS ────────────────────────────── A record → mullet WAN IP │ │ HTTPS :443 ▼ mullet (Caddy) ── TLS termination (Let's Encrypt cert via Cloudflare DNS-01) │ │ HTTP reverse proxy → 192.168.0.201:80 ▼ MetalLB VIP 192.168.0.201 ── LoadBalancer service: monitoring/grafana │ ▼ Grafana pod (inside cluster)

Components

Cloudflare DNS

Each service has an A record in Cloudflare pointing to mullet's WAN IP. Records are set to DNS-only (grey cloud — proxying disabled) so that Caddy receives the raw TCP connection and can terminate TLS itself. Cloudflare proxying is intentionally off: the cluster services are not routed through Cloudflare's CDN/WAF layer.

Cloudflare is also used as the ACME DNS provider. Caddy calls the Cloudflare API to create a temporary _acme-challenge TXT record to prove domain ownership during certificate issuance. This avoids any requirement for port 80 to be publicly reachable.

Caddy on mullet

Caddy is installed as a systemd service on mullet (Ubuntu 22.04). The configuration lives at /etc/caddy/Caddyfile. Caddy uses the caddy-dns/cloudflare plugin (compiled into the binary) to handle DNS-01 challenges automatically.

MetalLB

MetalLB assigns each Kubernetes LoadBalancer service a stable IP from the pool 192.168.0.200–192.168.0.230. These IPs are on the cluster LAN and are only reachable from mullet and other LAN hosts — they are never directly exposed to the internet. Caddy's reverse_proxy directives target these IPs.

Caddyfile structure

The global block enables the Cloudflare DNS plugin for all TLS certificates:

{
    dns cloudflare {env.CLOUDFLARE_API_TOKEN}
    debug
}

Each service block follows the same pattern:

grafana.verticon.com {
    reverse_proxy http://192.168.0.201:80   # MetalLB VIP for the service
    tls {
        resolvers 1.1.1.1 1.0.0.1           # use Cloudflare DNS to verify the DNS-01 challenge
    }
}

The tls { resolvers ... } block tells Caddy to use Cloudflare's public resolvers when verifying that the _acme-challenge TXT record has propagated before completing the certificate request.

Note: The Harbor entry adds request_body { max_size 0 } to allow unlimited image push sizes. Any service that accepts large uploads needs this override.

Adding a new service

  1. Assign a MetalLB IP. Pick the next free IP from the pool (192.168.0.200–230). Check current assignments with:
    kubectl get svc -A --field-selector spec.type=LoadBalancer
  2. Annotate the Kubernetes service. Add the MetalLB annotation to the service manifest: 
    metadata:
  annotations:
    metallb.universe.tf/loadBalancerIPs: "192.168.0.NNN"
  3. Add a Cloudflare DNS A record. In the Cloudflare dashboard for verticon.com, create an A record:
    Name: <service>  →  Content: <mullet WAN IP>
    Set to DNS-only (grey cloud, not proxied).
  4. Add a Caddyfile block on mullet at /etc/caddy/Caddyfile: 
    <service>.verticon.com {
    reverse_proxy http://192.168.0.NNN:80
    tls {
        resolvers 1.1.1.1 1.0.0.1
    }
}
  5. Reload Caddy:
    sudo systemctl reload caddy
    Caddy will automatically request a TLS certificate for the new hostname via DNS-01.
  6. Update docs/index.html — add the service to the Web UIs or Service Endpoints section.

Troubleshooting

Certificate not issued / renewal failed

  • Check Caddy logs: journalctl -u caddy -f
  • Confirm the CLOUDFLARE_API_TOKEN env var is set and the token has Zone DNS Edit permissions for verticon.com
  • Confirm the Cloudflare A record exists and is set to DNS-only (not proxied)
  • Caddy caches cert failures — after fixing, run sudo systemctl restart caddy to retry immediately

Service unreachable after adding Caddyfile block

  • Verify the MetalLB service has an external IP: kubectl get svc -n <ns> <name>
  • Test the LAN target directly from mullet: curl -v http://192.168.0.NNN:80
  • Confirm Caddy reloaded cleanly: sudo systemctl reload caddy && journalctl -u caddy -n 30

Validate Caddyfile syntax before reload

caddy validate --config /etc/caddy/Caddyfile
